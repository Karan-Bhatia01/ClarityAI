PROJECT OVERVIEW
================================================================================

ClarityAI is a full-stack data intelligence platform that automates the entire 
ML lifecycle. Users upload raw CSV files and the system guides them through:
  1. Data analysis & AI insights
  2. Preprocessing & EDA (Exploratory Data Analysis)
  3. On-demand chart interpretation
  4. Automated ML pipeline training

Tech Stack:
  - Backend: Flask (Python)
  - Database: MongoDB + GridFS (file storage)
  - Frontend: HTML/CSS/JS (Jinja2 templates)
  - Data Processing: Pandas, NumPy, Scikit-learn
  - ML Models: 9 parallel model families
  - LLM Integration: OXLO API (Mistral 7B) for insights + Vision API

USER JOURNEY - STEP BY STEP
================================================================================

STEP 1: HOME PAGE (Route: /)
================================================================================
File: templates/index.html + app.py - index() function
Backend: src/components/data_ingestion.py - DataIngestion class

What Happens:
  1. User lands on the upload page
  2. Page displays:
     - File upload form (CSV files only)
     - List of previously uploaded datasets
     - Preview of the latest dataset (first 5 rows)

Backend Flow:
  - DataIngestion.store_file(file) → Stores file in MongoDB GridFS
  - DataIngestion.get_all_filenames() → Retrieves all CSV filenames
  - DataIngestion.get_preview() → Reads latest file, returns first 5 rows + columns

Key Features:
  - Robust CSV parsing (tries 4 encodings, 3 separators, 2 bad_lines strategies)
  - File persistence in MongoDB GridFS (survives server restarts)
  - Shows previously uploaded datasets for quick re-analysis


STEP 2: DATASET ANALYSIS (Route: /info?filename=X)
================================================================================
File: templates/info.html + app.py - info_layer() function
Backend: src/agenticLayer/llm.py - AnalysisExplainer class

What Happens:
  1. User clicks "View Info" on a dataset
  2. System analyzes the dataset and returns:
     - Dataset Overview: shape, columns, nulls, duplicates
     - AI Insights: Summary, quality flags, column insights, recommendations, 
       uncertainty notes

Backend Flow:
  User clicks "View Info"
     ↓
  app.info_layer() receives filename
     ↓
  AnalysisExplainer(filename).run()
     ↓
  Loads CSV from MongoDB GridFS
     ↓
  Computes statistics (nulls, duplicates, describe, dtype)
     ↓
  Calls LLM with prompt: "Analyze this data and provide JSON with: summary, 
  quality_flags, column_insights, next_steps, uncertainty_notes"
     ↓
  LLM (Mistral 7B via OXLO) returns structured JSON
     ↓
  Renders info.html with analysis + ai_insights

Data Flow:
  - DataInfo.get_dataframe() → Fetches latest file from GridFS, parses as DF
  - DataInfo.dataset_analysis() → Computes shape, nulls, duplicates, stats
  - DataInfo.get_unique_column_values() → Gets sample unique values
  - AnalysisExplainer.run() → Combines stats, calls LLM, returns JSON

AI Insights Structure (JSON from LLM):
  {
    "summary": "plain text overview",
    "quality_flags": [
      {
        "column": "...",
        "severity": "high/medium",
        "issue": "...",
        "detail": "..."
      }
    ],
    "column_insights": [
      {
        "column": "...",
        "insight": "description"
      }
    ],
    "next_steps": [
      {
        "title": "...",
        "detail": "..."
      }
    ],
    "uncertainty_notes": "caveats and assumptions"
  }


STEP 3: PREPROCESSING + EDA (Route: /preprocessing?filename=X)
================================================================================
File: templates/eda_processing.html + app.py - preprocessing_inputs() function
Backend: src/components/eda_processing.py - DataPreprocessing class

What Happens (GET request):
  1. User clicks "Preprocess &amp; EDA" from info page
  2. System loads dataset and shows a form with:
     - Target column selector: dropdown of all columns
     - Columns to drop: checkboxes to exclude columns
     - Missing value strategy: per-column dropdowns 
       (Drop, Mean, Median, Mode, Forward Fill)
     - AI Suggestions: LLM recommends strategies for columns with nulls

Backend Flow (GET):
  /preprocessing GET request
     ↓
  DataPreprocessing() loads CSV from GridFS
     ↓
  If dataset has nulls:
    get_ai_insights() → LLM call for strategies
    Returns: {"col_name": "mean/median/mode/drop/forward_fill", ...}
     ↓
  Renders eda_processing.html with:
    - columns list
    - null counts per column
    - ai_strategy (LLM recommendations)
    - previous user selections (from session)

What Happens (POST request):
  1. User selects:
     - Target column
     - Columns to drop
     - Missing value strategy per column
  2. Clicks "Generate EDA Reports"
  3. System processes and shows charts

Backend Flow (POST):
  /preprocessing POST request with form data
     ↓
  Extract: target_column, columns_to_drop[], missing_[col_name]
     ↓
  Save to Flask session for later retrieval
     ↓
  preprocess_data(missing_value_strategy) 
    → applies strategies, drops columns, returns cleaned DataFrame
     ↓
  generate_eda_report(cleaned_df) 
    → creates charts (matplotlib) and returns as base64 images
    Examples:
      - Univariate: histograms, distributions
      - Bivariate: scatter plots, heatmaps
      - Trivariate: 3D plots (if applicable)
     ↓
  Returns dict: {"Chart Title": "image_b64_encoded", ...}
     ↓
  Renders preprocessing_result.html with:
    - cleaned dataset preview
    - all EDA charts (as base64 images)
    - buttons to analyze each chart on-demand


STEP 4: ON-DEMAND CHART ANALYSIS (Route: /analyse_chart - POST)
================================================================================
File: JavaScript in preprocessing_result.html calls this route
Backend: app.py - analyse_chart_route() function

What Happens:
  1. User clicks "Analyze" button on a chart
  2. Chart image (base64) is sent to backend
  3. System calls Vision API to explain the chart
  4. Analysis appears below the chart

Backend Flow:
  User clicks chart's "Analyze" button
     ↓
  JavaScript sends: {image_b64, chart_title}
     ↓
  analyse_chart_route() receives POST
     ↓
  Calls: analyse_chart(image_b64, chart_title, oxlo_api_key, ...)
     ↓
  Vision API analyzes image and returns:
  {
    "represents": "what this chart shows",
    "key_findings": ["finding 1", "finding 2"],
    "anomalies": ["anomaly 1"],
    "recommendations": ["action 1"]
  }
     ↓
  Returns JSON to frontend
     ↓
  JavaScript displays analysis below chart

Note: This is ON-DEMAND (not during preprocessing) to save API costs initially.


STEP 5: ML PIPELINE (Route: /ml?filename=X)
================================================================================
File: templates/ml_input.html + app.py - ml_pipeline_route() function
Backend: src/components/ml_pipeline.py - MLPipeline class

What Happens (GET request):
  1. User clicks "ML Pipeline" button
  2. Form shows dropdown to select target column

What Happens (POST request):
  1. User selects target column and clicks "Train Models"
  2. System trains 9 model families in PARALLEL

Models Trained (9 families):
  1. Logistic Regression (classification)
  2. Random Forest (classification & regression)
  3. Decision Tree (classification & regression)
  4. Gradient Boosting (classification & regression)
  5. XGBoost (if installed)
  6. LightGBM (if installed)
  7. KNN (classification & regression)
  8. Naive Bayes (classification)
  9. SVM (classification & regression)

Backend Flow:
  User selects target column on ml_input.html
     ↓
  /ml POST request
     ↓
  MLPipeline(filename, target_column).run()
     ↓
  load_dataframe_from_mongo(filename) → loads cleaned CSV from GridFS
     ↓
  Auto-detect problem type:
    - If target is numeric → Regression
    - If target is categorical → Classification
     ↓
  Feature Engineering:
    - Encode categorical columns (one-hot, ordinal)
    - Scale numeric features
    - Handle missing values
     ↓
  Train 9 models in parallel using ThreadPoolExecutor
    Each model: train / validate / compute metrics
     ↓
  Rank models by performance metric (accuracy for classification, R² for regression)
     ↓
  Save all models to MongoDB as .pkl files
     ↓
  Return to frontend:
    - problem_type: "classification" or "regression"
    - feature_plan: {"feature_engineering": "...description..."}
    - results: [{"model": "LogisticRegression", "accuracy": 0.85, ...}, ...]
    - best_model: {"name": "RandomForest", "metrics": {...}, ...}
    - rank_metric: "accuracy" or "r2_score"
    - mongo_doc_id: saves results to MongoDB
     ↓
  Renders ml_results.html with:
    - Ranked model comparison table
    - Best model details + confusion matrix / feature importance
    - All models available for download / deployment


================================================================================
DATABASE STRUCTURE (MongoDB)
================================================================================

clarityAI_database/
├── fs.files (GridFS metadata)
│   ├── _id: ObjectId
│   ├── filename: "my_data.csv"
│   ├── contentType: "text/csv"
│   ├── uploadDate: ISODate
│   └── length: file_size
│
├── fs.chunks (GridFS actual file bytes)
│   ├── files_id: references fs.files._id
│   ├── n: chunk number
│   └── data: binary chunk
│
├── ml_results (stored ML model results)
│   ├── _id: ObjectId
│   ├── filename: "my_data.csv"
│   ├── target_column: "result"
│   ├── problem_type: "classification"
│   ├── results: [model results array]
│   ├── best_model: {...}
│   └── timestamp: ISODate
│
└── model_cache (pickled trained models)
    ├── _id: ObjectId
    ├── filename: "my_data.csv"
    ├── model_name: "RandomForest"
    ├── model_pickle: binary
    └── timestamp: ISODate


================================================================================
COMPONENT RESPONSIBILITIES
================================================================================

1. src/components/data_ingestion.py (DataIngestion class)
   - Store files in MongoDB GridFS
   - Retrieve file list
   - Parse CSV robustly (handles encoding/separator issues)
   - Get dataset preview (first 5 rows)

2. src/components/data_info.py (DataInfo class)
   - Load dataset from GridFS as DataFrame
   - Compute basic statistics (shape, nulls, duplicates, describe)
   - Get unique values per column
   - Return analysis dict

3. src/agenticLayer/llm.py (AnalysisExplainer class)
   - Combines statistics + unique values
   - Calls OXLO LLM (Mistral 7B) with system prompt
   - Parses JSON response
   - Returns structured insights (summary, flags, recommendations)

4. src/components/eda_processing.py (DataPreprocessing class)
   - Get AI-suggested preprocessing strategies
   - Apply missing value handling (drop, mean, median, mode, forward fill)
   - Drop specified columns
   - Generate EDA charts (univariate, bivariate, trivariate)
   - Convert charts to base64 for HTML embedding
   - On-demand chart analysis via Vision API

5. src/components/ml_pipeline.py (MLPipeline class)
   - Auto-detect problem type (classification vs regression)
   - Feature engineering (encode, scale)
   - Train 9 model families in parallel
   - Compute metrics (accuracy, F1, precision, recall, R², MSE, MAE)
   - Rank models and select best
   - Save models to MongoDB
   - Generate confusion matrices + feature importance

6. src/utils.py
   - Load DataFrames from GridFS
   - LLM calls via OXLO API
   - Chart to base64 conversion (fig_to_b64)
   - Chart analysis (analyse_chart)
   - JSON parsing helpers

================================================================================
KEY DESIGN PATTERNS
================================================================================

1. Session Storage
   - User form selections stored in Flask session for persistence

2. On-Demand Analysis
   - Charts analyzed only when user clicks (not batch)
   - Saves API costs by avoiding unnecessary calls

3. Parallel Model Training
   - ThreadPoolExecutor trains 9 models simultaneously
   - Significantly reduces ML pipeline execution time

4. Base64 Image Embedding
   - Charts converted to base64 for HTML rendering
   - Allows displaying images without separate file requests

5. GridFS for Large Files
   - MongoDB GridFS handles file storage + retrieval
   - Supports files larger than 16MB (BSON document limit)

6. Robust CSV Parsing
   - Multiple encodings/separators tried progressively
   - Handles real-world messy CSV files

  - Jinja2 for template rendering

  1. Users upload raw CSV files
  2. System automatically analyzes and flags data quality issues
  3. Users configure preprocessing with AI guidance
  4. System generates exploratory charts and on-demand interpretations
  5. System trains multiple model families and ranks by performance
  6. Users get clear, actionable insights without writing code
